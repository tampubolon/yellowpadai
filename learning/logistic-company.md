## What I Built: Compass PaaS

Compass PaaS is a multi-tenant, cloud-native SaaS platform for logistics rate card management, built from the ground up on AWS. The system consists of:

- **3 FastAPI microservices** (`business-logic-service`, `integration-service`, `ma-layer-service`)
- **5 shared Python libraries** (auth, aws, core, db, logger) distributed as internal wheels
- **Full Terraform IaC** provisioning 20+ AWS resources across 3-tier VPC architecture
- **AI/LLM agent layer** (LangGraph + multi-agent orchestration with PostgreSQL checkpointing)
- **External integrations** (HubSpot, Monday.com, Zendesk, Slack, Google Drive)

## Platform Functionality

Compass PaaS is the backend platform powering a **logistics rate card management** product used by freight/3PL businesses. It handles the full lifecycle of pricing agreements between companies and their clients or suppliers.

### What the Product Does

Logistics companies operate with two sides of pricing:

- **Sell-side:** Rate cards offered to clients — what the company charges for fulfillment and last-mile delivery
- **Buy-side:** Rate cards received from suppliers — what the company pays for services

Compass manages both sides in a unified data model, enabling margin visibility and rate comparison across hundreds of client and supplier relationships.

### Business Logic Service — Core Domain

The primary service handles all data persistence and business rules across two domains:

**Sales Domain** (client-facing):
- Manage client companies and their account status
- Create and maintain sell rate cards with complex pricing tiers
- Model fulfillment items (warehouse handling) and last-mile items (delivery zones/ranges)
- Support bulk import of rate cards from Excel files uploaded to S3 (idempotent batch processing)
- Generate exports in Excel and PDF format

**Suppliers Domain** (supplier-facing):
- Manage supplier companies and relationships
- Ingest buy rate cards from suppliers (manual entry or batch import)
- Mirror the sell-side model for buy-side cost data

**Key capabilities:**
- Full CRUD with soft deletes and status tracking (draft, active, archived)
- Filtering by client, supplier, status, and date range
- Offset-based pagination for large datasets
- Idempotent S3 batch import — files can be reprocessed without creating duplicates
- OpenAPI/Swagger documentation auto-generated by FastAPI

### Integration Service — External Tools Layer

An adapter layer that connects Compass to the tools logistics operations teams already use:

| Adapter | Purpose |
|---------|---------|
| HubSpot | Sync client/contact data from CRM |
| Monday.com | Track onboarding tasks and project milestones |
| Zendesk | Link support tickets to client accounts |
| Slack | Push notifications for rate card approvals and alerts |
| Google Drive | Access documents shared by clients |

Each adapter implements consistent retry logic (exponential backoff), rate limit handling, idempotency keys, and typed error responses — so the rest of the platform treats all integrations uniformly regardless of the underlying API's quirks.

### MA Layer Service — AI Agent Orchestration

The intelligence layer built on **LangGraph** and **LangChain**, providing multi-step AI workflows:

- **Extraction pipeline:** Parse and extract structured rate card data from unstructured documents (PDFs, Excel files) using LLM calls
- **Rate validation:** Automated cross-checking of buy vs. sell rates to flag margin issues or data inconsistencies
- **Conversational agents:** Stateful multi-turn conversations backed by PostgreSQL checkpoints — agents can be interrupted and resumed without losing context
- **Multi-provider LLM support:** Abstracted provider clients for OpenAI (GPT-4), Anthropic (Claude 3), Cohere, Mistral, and Google (Gemini) — the platform can swap or A/B test providers without application changes

LangSmith integration provides full trace observability for agent runs, including token usage, tool calls, and latency per step.

### Shared Platform Libraries

Rather than duplicating concerns across services, five internal Python libraries are distributed as wheels and installed at Docker build time:

| Library | What it provides |
|---------|-----------------|
| `compass-auth` | JWT validation (RS256 for Cognito, HS256 for dev), mock auth strategy, request correlation middleware |
| `compass-aws` | boto3 wrappers for S3, Secrets Manager, SSM, SES, ECR, ECS |
| `compass-core` | Pydantic v2 domain models, common exceptions, CORS + error handler middleware |
| `compass-db` | SQLAlchemy 2.0 async engine, session factory, dependency injection for FastAPI |
| `compass-logger` | structlog-based structured logging, X-Request-ID propagation, JSON formatter for production |

This wheel-based approach means a single change to `compass-auth` propagates to all three services on the next build — no copy-paste, no drift.

### Control Center — Admin CLI

A Python CLI for operational tasks that must happen outside the API:

- `register_admin.py` — provisions a new admin user in both Cognito (sets password, assigns group) and the database in a single transactional operation with rollback on failure
- `delete_admin.py` — removes admin users with the same dual-system cleanup

### Data Model Highlights

The PostgreSQL schema models the core logistics pricing domain:

```
clients ──< sell_rate_cards ──< sell_fulfillment_items
                           └──< sell_last_mile_items ──< sell_last_mile_ranges

suppliers ──< buy_rate_cards ──< buy_fulfillment_items
                             └──< buy_last_mile_items ──< buy_last_mile_ranges
```

Status enums (`draft`, `active`, `archived`) and soft deletes (`deleted_at`) give full audit history. Alembic manages all schema migrations with environment-specific execution paths (local, staging via bastion tunnel, production via bastion tunnel).

### Authentication Deep Dive

The `compass-auth` library implements a **pluggable strategy pattern** so the same middleware works in all environments without code changes:

```
                    AuthenticationMiddleware
                           │
                    CompositeStrategy
                    ┌──────┴───────┐
                    │              │
              JwtStrategy    MockStrategy
                    │              │
              JwtAuthenticator   (dev/test only)
              ┌─────┴──────┐
              │            │
          RS256Decoder  HS256Decoder
          (production)   (local dev)
```

**Production flow (RS256):**
1. Client sends `Authorization: Bearer <token>` obtained from Cognito login
2. Middleware calls `JwtAuthStrategy.authenticate(request)`
3. RS256Decoder fetches Cognito's JWKS public keys (cached), verifies signature, validates `exp`/`iss`/`aud` claims
4. Claims mapped to `AuthenticatedUser(id, cognito_username, email, role, token_age_seconds, request_id)`
5. User stored in `request.state.context` — downstream dependencies read it without touching the token again
6. On failure: `AuthError` → 401 JSON response with `X-Request-ID` for correlation

**Local dev flow (HS256):** Mock strategy bypasses Cognito entirely, accepts a pre-shared secret token — no network dependency during development.

### AI Agent Flow (MA Layer)

The MA Layer service is built around a **compiled LangGraph state machine** that runs as a streaming HTTP endpoint:

```
Client                     MA Layer Service                    PostgreSQL
  │                              │                                  │
  │  POST /chat/stream-response  │                                  │
  │  { user_input, conv_id }     │                                  │
  │  (SSE, keep-alive)           │                                  │
  │─────────────────────────────▶│                                  │
  │                              │ validate_conversation_id()       │
  │                              │ SELECT conversation WHERE        │
  │                              │   id=conv_id AND user_id=me      │
  │                              │─────────────────────────────────▶│
  │                              │◀─────────────────────────────────│
  │                              │                                  │
  │                              │ _resolve_input()                 │
  │                              │ aget_state(thread_id=conv_id)    │
  │                              │ ← reads last checkpoint          │
  │                              │─────────────────────────────────▶│
  │                              │◀─────────────────────────────────│
  │                              │                                  │
  │                              │ graph.astream_events(input_data) │
  │                              │                                  │
  │                              │ ┌────────────────────────────┐   │
  │                              │ │  LangGraph State Machine   │   │
  │                              │ │                            │   │
  │                              │ │  user_input_node           │   │
  │                              │ │       ↓                    │   │
  │                              │ │  executor_node             │   │
  │                              │ │  (selects next agent)      │   │
  │                              │ │       ↓                    │   │
  │                              │ │  [subgraph: extraction /   │   │
  │                              │ │   validation / chat]       │   │
  │                              │ │       ↓                    │   │
  │                              │ │  LLM call (streaming)      │   │
  │                              │ └────────────────────────────┘   │
  │                              │                                  │
  │  ← SSE chunk (text)          │ on_chat_model_stream event       │
  │  ← SSE chunk (text)          │ yield JSON: {"type":"text",      │
  │  ← SSE chunk (text)          │   "content": "..."}              │
  │                              │                                  │
  │                              │ checkpoint saved after each node │
  │                              │─────────────────────────────────▶│
  │                              │  checkpoints / checkpoint_blobs  │
  │                              │  / checkpoint_writes tables      │
  │                              │                                  │
  │  [stream ends]               │ update_conversation_timestamp()  │
  │                              │─────────────────────────────────▶│
```

**Key design decisions visible in the code:**

- `_resolve_input()` checks `state.next` — if the graph is paused at a human-input interrupt node, the next message resumes it (`human_response`) instead of starting a new turn (`question`). This enables **multi-turn interruption** — the agent can ask a clarifying question and wait.
- On `asyncio.CancelledError` (client disconnect): the partial response is saved to the checkpoint with a cancellation note, so the conversation history stays coherent even for aborted streams.
- On `GraphRecursionError`: the agent hit its recursion limit (25 steps) — the graph state is reset and an intelligible message is streamed to the client rather than surfacing a raw exception.
- Checkpoints use `AsyncPostgresSaver` — the same PostgreSQL database stores both business data and full agent state (messages, current plan, next agent). Deleting a conversation explicitly cleans `checkpoints`, `checkpoint_blobs`, and `checkpoint_writes` tables in order (FK-aware).

### Service-to-Service Communication

Services call each other over the **Cloud Map private DNS namespace** — no public network hop:

```
Business Logic Service (ECS task)
         │
         │  httpx async HTTP client
         │  http://wayfindr-compass-integration
         │      .wayfindr-compass-{env}.local:5001
         │
         ▼
Integration Service (ECS task, same VPC)
         │
         │  httpx with retry + exponential backoff
         │  (external network only for this hop)
         ▼
    External API (HubSpot, Slack, etc.)
```

The `integration_service_client.py` in BLS and `business_logic_client.py` in MA Layer are thin typed HTTP clients — the service URLs are injected via environment variables resolved from Cloud Map DNS at deploy time.

M2M authentication between services uses Cognito OAuth2 client credentials — each service has its own app client. Secrets are rotated via `update-m2m-secrets.sh` and stored in Secrets Manager.

### Database Session Lifecycle

Each HTTP request gets its own isolated async database session:

```python
# compass-db library — simplified
async def get_db_session() -> AsyncGenerator[AsyncSession, None]:
    async with session_factory() as session:
        try:
            yield session          # endpoint runs here
            await session.commit() # auto-commit on success
        except Exception:
            await session.rollback()
            raise
        finally:
            await session.close()
```

SQLAlchemy's async connection pool (asyncpg) maintains a pool of persistent connections to Aurora — individual requests don't pay the TCP handshake cost. Aurora Serverless v2 scales the compute layer transparently; the pool sees it as a stable host.

### Containerization & Multi-Stage Docker Builds
All 3 services use an optimized **multi-stage build pattern**:

- **Stage 1 (Builder):** `python:3.13-slim` — installs gcc/libpq-dev, builds shared library wheels, installs service deps to `/install` prefix
- **Stage 2 (Runtime):** `python:3.13-slim` — copies only `/install` from builder, no build tools in final image

Result: lean production images with no build toolchain artifacts. Business Logic Service additionally installs WeasyPrint system deps (PDF generation) in a targeted, minimal way.

### Networking (VPC, Security Groups, TLS, DNS)
Designed and implemented a **3-tier VPC architecture** (`10.1.0.0/16`):

- **Tier 1 (Public):** ALB + WAF, accepts 80/443, redirects HTTP → HTTPS
- **Tier 2 (Private):** ECS Fargate services, accessible only from ALB on service ports
- **Tier 3 (Database):** Aurora PostgreSQL cluster, accessible only from Tier 2 + Bastion on port 5432
- **Security Groups:** Least-privilege ingress rules between tiers, no direct internet access from private/database tiers
- **TLS:** ACM wildcard certificate (`*.compass.wayfindr.io`) with automated Route53 DNS validation
- **DNS:** Route53 hosted zone for `compass.wayfindr.io`, A records aliased to ALB

### Secrets Management (AWS Secrets Manager + SSM)
Implemented a dual-layer secrets architecture:

- **AWS Secrets Manager:** Sensitive credentials (DB master password, app user passwords, service API keys) with automatic rotation
- **AWS Systems Manager Parameter Store:** Non-sensitive config (endpoints, ARNs) under namespace `/{app_name}/{environment}/...`
- **ECS integration:** Secrets injected as environment variables from Secrets Manager ARNs in task definitions — containers never touch raw credentials directly
- **Bastion operations:** Python script (`bastion_tunnel.py`) fetches SSM ARNs to resolve Secrets Manager values for manual operational tasks

### PostgreSQL Administration
Built full Aurora PostgreSQL v17 Serverless v2 infrastructure including:

- Serverless v2 scaling (0.5–1 ACU), auto-pause, 7-day backup retention
- Schema design for multi-tenant rate card management (20+ tables)
- Alembic migration pipeline with remote execution via bastion SSM tunnel
- Separate master user (Secrets Manager rotation) and application user (least-privilege)
- PostgreSQL also powers **LangGraph agent checkpoints** for stateful multi-agent workflows in the MA Layer service
- Local dev → staging → production migration workflow via `make db-upgrade-*` targets

## Future Improvement: Self-Hosted LLM on Kubernetes GPU Node Pool

Currently the MA Layer service delegates all LLM inference to external APIs (OpenAI, Anthropic, etc.). The future goal is to run a **self-hosted LLM** — eliminating external API costs, removing data egress concerns, and giving full control over the model weights. This requires two distinct pipelines: a **training pipeline** (fine-tune the base model on domain data) and an **inference pipeline** (serve the trained model to the MA Layer at runtime).

Both pipelines share the same EKS cluster but use dedicated **GPU node groups** with taints/tolerations to ensure GPU workloads never land on CPU nodes and vice versa.

---

### Node Pool Design

```
EKS Cluster
├── cpu-node-group           (general workloads: BLS, Integration, MA Layer app logic)
│   ├── instance: m7i.2xlarge (8 vCPU, 32 GB RAM)
│   └── no GPU, no taint
│
├── gpu-training-node-group  (batch fine-tuning jobs, ephemeral)
│   ├── instance: p4d.24xlarge (8× NVIDIA A100 40GB, NVLink)
│   │            or p5.48xlarge (8× NVIDIA H100 80GB, NVLink + EFA)
│   ├── AMI: Amazon EKS-optimized Accelerated AMI (al2-nvidia)
│   ├── label: node-role=gpu-training
│   ├── taint: nvidia.com/gpu=training:NoSchedule
│   └── Cluster Autoscaler: min=0, max=4 nodes (scale to 0 when idle)
│
└── gpu-inference-node-group (always-on model serving)
    ├── instance: g6.2xlarge  (1× NVIDIA L4 24GB)   — preferred for inference
    │            or g5.2xlarge (1× NVIDIA A10G 24GB) — alternative
    ├── AMI: Amazon EKS-optimized Accelerated AMI (al2-nvidia)
    ├── label: node-role=gpu-inference
    ├── taint: nvidia.com/gpu=inference:NoSchedule
    └── Cluster Autoscaler: min=1, max=10 nodes (KEDA-driven scale-out)
```

#### 1b. Install NVIDIA Device Plugin (DaemonSet)

The NVIDIA device plugin exposes each GPU on a node as an allocatable resource (`nvidia.com/gpu: 1`) so the Kubernetes scheduler can assign GPU pods correctly. Without it, pods request GPUs but the scheduler has no awareness of them.

```bash
# Install via Helm — runs as DaemonSet on every GPU node
helm repo add nvdp https://nvidia.github.io/k8s-device-plugin
helm repo update

helm upgrade --install nvidia-device-plugin nvdp/nvidia-device-plugin \
  --namespace kube-system \
  --set failOnInitError=false
```

The DaemonSet uses a `nodeSelector` targeting nodes with NVIDIA drivers already installed (provided by the AL2 GPU AMI). It mounts the host's NVIDIA runtime socket and registers each physical GPU as a schedulable resource.

```
GPU Node (p4d.24xlarge)
├── Host OS: Amazon Linux 2 + NVIDIA driver 535.x (baked into AL2 GPU AMI)
├── Container runtime: containerd + nvidia-container-toolkit
└── DaemonSet pod: nvidia-device-plugin
    └── registers: nvidia.com/gpu = 8  (advertised to kube-apiserver)
```

#### KEDA ScaledObject (queue-depth autoscaling)

```yaml
# k8s/inference/scaledobject.yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: vllm-scaledobject
  namespace: ml-inference
spec:
  scaleTargetRef:
    name: vllm-server
  minReplicaCount: 1
  maxReplicaCount: 10
  cooldownPeriod: 300           # 5 min scale-down cooldown — avoid flapping
  pollingInterval: 15
  triggers:
    - type: prometheus
      metadata:
        serverAddress: http://prometheus.monitoring.svc:9090
        metricName: vllm_num_requests_waiting
        query: sum(vllm_num_requests_waiting{namespace="ml-inference"})
        threshold: "5"          # scale out when > 5 requests queued per replica
```

**Key tools:** vLLM (inference server + OpenAI-compatible API), KEDA (event-driven HPA), Argo CD (GitOps model rollout), Prometheus + Grafana (GPU metrics dashboard), AWS Cluster Autoscaler (node-level scale-out).

### End-to-End Flow Summary

```
[New domain data available]
         │
         ▼
CronJob: export Compass DB + S3 raw files → s3://datasets/raw/
         │
         ▼
Preprocessing Job (cpu-node-group)
  HuggingFace datasets: clean, tokenize, format, split
  → s3://datasets/processed/{run_id}/
         │
         ▼
Kubeflow PyTorchJob (gpu-training-node-group, p4d.24xlarge)
  torchrun (4 nodes × 8 GPUs = 32× A100)
  SFTTrainer + QLoRA/PEFT → adapter_model.safetensors
  → s3://models/compass-llm/{run_id}/
  → MLflow: loss, perplexity logged every epoch
         │
         ▼
Eval Job (cpu-node-group)
  lm-evaluation-harness → domain benchmarks
  → results logged to MLflow
         │
         ├── FAIL: run discarded, retrain with adjusted hyperparams
         │
         └── PASS: MLflow tags run "production"
                   webhook updates Git kustomization.yaml
                   │
                   ▼
         Argo CD detects Git diff
         → rolling update: vLLM Deployment (gpu-inference-node-group)
           init container: s3 sync → merge adapter → /model-cache/merged/
           vLLM loads merged model, readiness probe passes
           old pods drained, new pods serve traffic
                   │
                   ▼
         MA Layer Service (cpu-node-group)
         OPENAI_BASE_URL=http://vllm-service:8000/v1
         LangGraph → /v1/chat/completions → vLLM → L4 GPU
         LangSmith traces: token usage, latency, tool calls
                   │
                   ▼
         KEDA: vllm_num_requests_waiting > 5
         → Cluster Autoscaler adds g6.2xlarge nodes
         → new vLLM pods scheduled, init container pulls model
```

### Why This Architecture

| Concern | Decision | Reason |
|---------|----------|--------|
| Fine-tune vs train from scratch | QLoRA fine-tune (PEFT) | 4-bit quantization fits 7B model on A100; fraction of full-precision training cost |
| Distributed training across nodes | Kubeflow PyTorchJob + torchrun + EFA | NCCL over EFA achieves near-linear scaling on p4d; avoids bottleneck at rank 0 |
| vLLM as inference server | OpenAI-compatible REST API | MA Layer zero code change; continuous batching + PagedAttention maximize L4 throughput |
| Separate training vs inference node pools | Different instance types + taints | p4d.24xlarge for burst training (scale to 0); g6.2xlarge for always-on serving (cost-efficient) |
| KEDA over standard HPA | Queue-depth metric from Prometheus | CPU/memory HPA is blind to inference backpressure; pending request count is the true saturation signal |
| Argo CD for model rollout | GitOps: model version in kustomization.yaml | Same rollout/rollback discipline as application code; full audit trail in Git |
| LoRA adapter stored separately from base model | S3 versioned objects | Base model (~14 GB) downloaded once and cached; only 200–500 MB adapter swapped per version |
| AL2 GPU AMI | EKS-optimized Accelerated AMI | NVIDIA drivers + nvidia-container-toolkit pre-baked; no driver bootstrap on node startup |
